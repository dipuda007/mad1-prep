Q&A: The Event-booking App Itself (30 questions)

⚠️ Note: examples on this page reference EventHub, the demo app built in this course — not any term's official project. Map every pattern to YOUR statement and never copy code into your submission (plagiarism = U grade).

Questions aimed at YOUR code. Answers reference exact files/lines — open the code while drilling these.

1. Walk me through your project structure.

app.py — Flask config + all route functions (controllers) + setup_database() seeding; models.py — 4 tables; templates/ — 12 Jinja pages sharing base.html; static/style.css — my CSS; requirements.txt — Flask, Flask-SQLAlchemy; instance/eventhub.db — SQLite file created at first run.

2. What did you understand from the problem statement?

Event-booking groups coordinate manually (spreadsheets/calls) → hard to track bookings, prevent overbooking, keep history. Build a three-role web app: admin runs the system, staff manage assigned events, attendees book — with role-based access and booking integrity rules.

3. Explain your models and relationships.

User (all 3 roles via role column; approved + blacklisted flags), StaffProfile (1-1 with User via user_id FK + uselist=False), Event (details + status + staff_id FK → the assigned staff User), Booking (user_id + event_id FKs, status, payment_status, booking_date). 1-N: user→bookings, event→bookings, staff→events.

4. Why one User table for three roles instead of three tables?

Login is a single lookup, no duplicated auth columns, role column switches behaviour. Staff-only extra fields go in the separate StaffProfile (keeps User lean, demonstrates 1-1).

5. How is the admin created? Why no admin registration?

setup_database() runs at startup: create_all(), then inserts admin (username admin, hashed password) only if no admin exists. Statement requires admin to pre-exist programmatically; register form only offers user/staff.

6. How does registration differ for staff vs user?

Same form; role dropdown. Staff get approved=False (login blocked until admin approves) and a StaffProfile row with phone. Users get approved=True immediately.

7. Explain your login flow line by line.

Look up user by username → if missing or check_password_hash fails: flash + back (one vague message so attackers can't tell which was wrong) → blacklisted? blocked → staff and not approved? blocked → store user_id + role in session → redirect to index which routes by role to /admin, /staff or /user.

8. How do users get to different dashboards?

index() reads current_user(); role == admin → admin_dashboard, staff → staff_dashboard, else user_dashboard. The navbar also renders different links per session role.

9. How is dashboard access restricted?

Every route starts with the guard: current_user() exists and has the right role, else redirect to login. Staff routes add ownership: event.staff_id == user.id.

10. What is current_user()?

My helper: if 'user_id' in session, fetch the User row (User.query.get), else None. One place to answer "who is logged in".

11. How does the admin dashboard get its counts?

A stats dict of four .count() queries — Event.query.count(), User filtered by role for users/staff, Booking.query.count() — passed to the template's stat cards.

12. Explain add event vs edit event.

Both render the same event_form.html. Add: GET shows blank form (event=None), POST builds an event from form fields (ints converted) and add+commit. Edit: get_or_404 loads it, POST overwrites fields, commit only — the object is already in the session.

13. How is staff assigned to an event?

The form's staff dropdown lists approved, non-blacklisted staff; the chosen id lands in event.staff_id. Empty choice stores None (NULL FK). Reassigning = edit event, choose another.

14. Why is delete a POST with a loop over bookings?

GET must never modify data (crawlers/prefetch would delete events!), so the delete button is a mini form. Bookings reference the event via FK, so children are deleted first — manual cascade, visible in code.

15. How does approve work?

POST /admin/approve/<id> → get_or_404 → person.approved = True → commit. Button rendered only for unapproved staff ({% if not person.approved %}).

16. How does blacklist work? Blacklist vs delete?

toggle: person.blacklisted = not person.blacklisted. Login checks it and blocks. Blacklist preserves the account + history and is reversible; delete would orphan bookings and lose history — that's why I blacklist instead of deleting people.

17. Explain your search (admin).

GET /admin/search?q=... → events by name ilike, people by name ilike; if q is all digits, also try Event.query.get / User.query.get and append if not already found → results template with two tables. Name OR id, case-insensitive, substring.

18. Explain the user search/filter.

GET form feeds request.args. Start from Event.query.filter_by(status='Open'); add filter_by(category=...) if chosen; add location ilike if typed; .all(). Chaining filters = AND conditions.

19. Recite the booking guards.

(1) event.status != 'Open' → reject; (2) available_slots <= 0 → reject (overbooking); (3) existing Booking with same user_id+event_id+status='Booked' → reject (duplicate). Then create booking with today's date, slots -= 1, one commit.

20. Why does the duplicate check filter status='Booked'?

So a cancelled booking doesn't block rebooking. Deliberate: cancel → slot returned → user may book again; history keeps both rows.

21. How do you prevent overbooking exactly?

Slot counter on Event: decremented on each booking inside the same commit, incremented on cancel; booking route rejects when <= 0. Plus staff can raise/lower slots.

22. How does cancel work? Why not delete the booking row?

POST → get_or_404 → ownership check (booking.user_id == session user) → only if status 'Booked': set 'Cancelled', slots += 1, commit. Row kept: history must be complete — status change instead of deletion.

23. How do staff see only their events?

Event.query.filter_by(staff_id=user.id) in staff_dashboard — the query itself is the restriction, not the template.

24. How is "only assigned staff can manage" enforced?

Ownership checks in every staff action: staff_update_event checks event.staff_id != user.id; staff_update_booking checks booking.event.staff_id != user.id (via backref); participants blocks staff on non-assigned events (admin allowed).

25. What are your event/booking status lifecycles?

Event: Pending → Approved → Open → Closed → Completed (admin sets any; staff move among Approved/Open/Closed/Completed; booking allowed only in Open). Booking: Booked → Cancelled (user/staff) or → Completed (staff).

26. Where does the booking count on the staff card come from?

A counts dict built in the route: for each assigned event, Booking.query.filter_by(event_id, status='Booked').count() — cancelled ones excluded on purpose.

27. What is payment_status? There's no payment gateway…

A field from the milestone's Booking spec. No real gateway (out of scope), so it defaults to 'Paid' at booking; the column exists so payment state can be tracked/extended. Honest answer, better than pretending.

28. What challenges did you face? (soft question — prepare a real one)

Good honest options: understanding how session keeps login across stateless HTTP; getting relationships/backrefs right (staff vs booking user both FK to User); remembering create_all doesn't alter tables (deleted the DB during development after model changes); slot arithmetic staying consistent across book/cancel/staff-cancel paths.

29. What would you improve with more time?

Chart.js summary charts, JSON API endpoints, Flask-Login, CSRF tokens via Flask-WTF, pagination for long tables, date validation (end after start), profile pictures. Naming concrete extensions shows awareness without over-claiming.

30. Rate your project / is this your field of interest? (they actually ask)

Be honest + grounded: "It meets every core requirement and I understand each line; UI is simple by choice. I'd rate it a solid core implementation with room for the optional enhancements." Interest: answer truthfully — proctors respect honesty.
