Lesson 9: Jinja2 Templates

Goal: send real HTML pages with Python data inside. Second-biggest viva topic after the database.

What is Jinja2?

The template engine bundled with Flask. A template is an HTML file with placeholders; Flask fills the placeholders with your Python data before sending it to the browser. Two delimiters to know:

SyntaxPurposeExample
{{ ... }}print a value into the page{{ event.name }}
{% ... %}logic: if, for, extends, block{% for t in events %}

render_template

Templates must live in a folder literally named templates. Flask finds it automatically (that's what Flask(__name__) was for).

# app.py
from flask import Flask, render_template

app = Flask(__name__)

@app.route('/')
def home():
    events = ['Music Night', 'Kedarkantha', 'Valley of Flowers']
    return render_template('home.html', events=events, count=len(events))
<!-- templates/home.html -->
<h1>We have {{ count }} events</h1>
<ul>
    {% for t in events %}
        <li>{{ t }}</li>
    {% endfor %}
</ul>

Note: every {% for %} needs {% endfor %}, every {% if %} needs {% endif %} — Jinja can't rely on indentation like Python.

if inside templates

{% if event.available_slots <= 0 %}
    <p>Sold out!</p>
{% else %}
    <p>{{ event.available_slots }} slots left</p>
{% endif %}
"Write a Jinja for loop / if" is a REAL live viva task (often in Notepad!). Practice writing from memory: {% for x in items %} {{ x }} {% endfor %} and {% if ... %} {% else %} {% endif %}. One real question: given a = [1,2,3], print elements greater than 1 → {% for n in a %}{% if n > 1 %}{{ n }}{% endif %}{% endfor %}

Template inheritance — the big one

Every page shares the navbar, Bootstrap link, flash messages. Instead of repeating that in 12 files, write it ONCE in base.html with holes, and let pages fill the holes.

<!-- templates/base.html -->
<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}Event-booking App{% endblock %}</title>
    <link href="...bootstrap.min.css" rel="stylesheet">
</head>
<body>
    <nav>...navbar shared by every page...</nav>
    <div class="container">
        {% block content %}{% endblock %}
    </div>
</body>
</html>
<!-- templates/login.html -->
{% extends 'base.html' %}
{% block title %}Login{% endblock %}
{% block content %}
    <h1>Login</h1>
    <form>...</form>
{% endblock %}
"What is template inheritance? Show it in your code." — a child template {% extends 'base.html' %} and overrides the {% block %} holes; common layout lives in one file (DRY — Don't Repeat Yourself). In your project: base.html holds navbar + Bootstrap + flash messages; all 12 pages extend it. Point at any template's first line.

url_for — never hardcode URLs

<a href="{{ url_for('login') }}">Login</a>
<a href="{{ url_for('edit_event', event_id=event.id) }}">Edit</a>
<link href="{{ url_for('static', filename='style.css') }}" rel="stylesheet">
"What goes inside url_for?" — the function name of the route (not the URL, not the template name!). url_for('login') builds "/login" from the function def login(). Extra arguments fill dynamic URL parts.

"url_for vs redirect vs render_template?" — url_for only builds a URL string; redirect(url) sends a 302 response telling the browser "go to this URL instead" (new request happens); render_template fills a template and returns the HTML in this response. Typical combo after a form: redirect(url_for('dashboard')).

Try it

Build a two-page app (home + about) sharing one base.html with a navbar linking both pages via url_for. Pass a list of 3 events to home and print them as a Bootstrap table with a for loop.
