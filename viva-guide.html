Viva Guide — how it actually goes

Distilled from the official project document, the viva checklist, and ~85 pages of real student reports of past vivas.

Both passed → L1+L2 marks sum to your grade. Not booking a slot when eligible = U (L1) / E (L2). Rescheduling only for medical reasons with proof; your-side cancellation costs ₹1000.

The standard Level 1 script (from dozens of real reports)

  1. Show student ID card (keep it open in a browser tab).
  2. Show GitHub repo with MADI-cs2003 as collaborator (keep the page open).
  3. Download YOUR submission from the portal live (or verify checksum) → unzip → run. Must be running within 10 minutes or you fail.
  4. Demo all core functionality (use the Build 6 story — it covers every rubric point).
  5. 1–2+ theory questions (the Q&A pages here cover ~95% of what's ever asked).
  6. A live code change (see the Code Tasks page).

Camera setup (non-negotiable)

Join the meet from phone AND pc. Phone camera positioned to show your hands, face and laptop screen; one device muted. Set this up BEFORE the viva starts — setup time counts inside the 10-minute rule. Join the link 10 minutes early. If the schedule email hasn't come, it arrives 12–24h before the slot.

Have open before it starts

  • ID card (browser tab) · GitHub repo page · project submission portal page
  • VS Code with nothing AI-related installed/visible — AI tools in the editor during viva = not allowed
  • A text file with your run commands (pip install -r requirements.txt, python app.py)
  • Notepad — proctors often ask you to write routes/loops there
  • Dependencies pre-installed; internet working (Bootstrap loads from CDN)

Malpractice — the real danger list

  • Complex project + can't answer basics = flagged. This is why your app is deliberately simple. 56 students got U for exactly this; 45 got U in ALL courses + 2-term ban.
  • Can't write simple logic or modify your own code when asked = flagged.
  • Rude/argumentative with the examiner = removed + U. Stay polite even if they're wrong.
  • Declared AI usage ≠ problem, IF you demonstrate understanding. Undeclared or ununderstood = problem.
  • "Code was on another laptop" is explicitly not accepted — develop and viva on the SAME machine.

Winning behaviours (straight from student reports)

  • Talk continuously while demoing. One student: "I was the one talking for 30 of the 33 minutes — he only said Perfect." Narrate: "this is my login route, it checks the hash, then stores user_id in session…" The more you explain unprompted, the fewer questions come.
  • Take a breath before answering; proctors consistently give time.
  • If a change breaks, don't panic — restart the server first (fixes 90% of "my change isn't showing"), and remember a stale database is the other 10% (delete instance/eventhub.db).
  • If you don't know, say what you DO know honestly. Bluffing invites deeper digging.
  • At the end, ask for feedback — multiple reports say proctors respond warmly to it.
  • If your proctor ID is shared in student groups, look up that proctor's past questions — "80% of questions repeat per proctor" per multiple reports. The Q&A pages here cover the union of all of them anyway.

Your 60-second project pitch (memorize, it opens every viva)

"I built an eventHub demo application using Flask with Jinja2 templates, SQLite through the Flask-SQLAlchemy ORM, and Bootstrap for the UI — the mandated stack, MVC pattern: models.py is the Model, templates are the View, the routes in app.py are the Controllers. There are three roles. Admin is pre-created programmatically and manages events, approves staff, assigns them to events, can blacklist anyone, and sees every booking. Staff register but need approval; they manage only their assigned events — slots, status, participants. Users register, search open events by category and location, and book — with duplicate bookings and overbooking blocked in the backend. Booking history is never deleted; statuses track the full lifecycle."
