Q&A: SQL, SQLite & ORM (50 questions)

The single most-examined area. Proctors reportedly spend 30+ minutes on models alone.

Database concepts

1. What is a database? DBMS?

Organized persistent storage of data. DBMS = the software managing it (SQLite, MySQL, PostgreSQL). Relational DBMS = data in tables with relationships.

2. Why SQLite? SQLite vs MySQL?

SQLite: serverless, zero-config, single file, ideal for local single-user apps — and the only DB the project permits. MySQL: separate server, concurrent users, networked, production scale.

3. Structured vs unstructured data/database?

Structured: fixed schema of rows/columns (SQL). Unstructured: no fixed shape — text, images, video. Semi-structured: JSON/XML (NoSQL DBs like MongoDB).

4. What is a schema?

The structure definition: tables, columns, types, constraints, relationships. Mine: User, StaffProfile, Event, Booking + their FKs (drawn as the ER diagram in my report).

5. What is an ER diagram?

Entity-Relationship diagram — boxes for tables/entities with attributes, lines for relationships with cardinality (1-1, 1-N, N-M).

6. Primary key? Can it be NULL?

Column uniquely identifying each row; NOT NULL and unique by definition; one per table. All my tables use an auto-incrementing integer id.

7. Foreign key?

Column holding another table's primary key to create a relationship — booking.user_id → user.id, booking.event_id → event.id, event.staff_id → user.id, staffprofile.user_id → user.id.

8. Primary vs unique key?

Both forbid duplicates. Primary: the row identifier, exactly one, never NULL. Unique: any number allowed, may allow NULL. username is unique but id is the primary key.

9. Composite key?

A primary key made of multiple columns together (e.g. (user_id, event_id) could identify a booking pair). I used surrogate integer ids instead — simpler and allows rebooking history.

10. Types of constraints?

PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, DEFAULT, CHECK. In SQLAlchemy: primary_key=True, ForeignKey(), unique=True, nullable=False, default=.

11. What is NULL? NULL vs NOT NULL?

NULL = absence of a value (not 0, not ""). NOT NULL constraint forbids it. event.staff_id is NULL until admin assigns staff — my "Not assigned" display checks that.

12. Relationship types + examples from YOUR app?

1-1: User↔StaffProfile. 1-N: User→Bookings, Event→Bookings, staff User→Events. N-M: none needed — would require a junction table of FK pairs.

13. Convert 1-N to N-M?

Remove the single FK; create an association table with both FKs (e.g. event_staff(event_id, staff_id)) — each row is one pairing; in SQLAlchemy use the secondary= parameter.

14. What is normalization?

Structuring tables to eliminate redundant data (each fact stored once). E.g. bookings store event_id, not the event's name/location — those live once in event. Levels are normal forms (1NF: atomic values; 2NF/3NF: attributes depend on the key, nothing else).

15. What is an index?

A lookup structure making searches on a column fast (like a book index), at the cost of storage and slower writes. Primary keys are indexed automatically.

16. What is a transaction / ACID?

A group of operations that succeed or fail together. ACID: Atomicity (all-or-nothing), Consistency (rules preserved), Isolation (parallel transactions don't interfere), Durability (committed = permanent). My booking = insert + slot decrement in ONE commit — atomic.

17. What is a view in SQL?

A saved SELECT query that acts like a virtual table. Not used in my project — Jinja templates are my "views" in the MVC sense (different concept, same word: know both).

18. DELETE vs TRUNCATE vs DROP?

DELETE removes rows (can filter with WHERE, row-by-row, can roll back). TRUNCATE empties the entire table fast (no WHERE; SQLite doesn't have it — DELETE FROM table serves). DROP removes the table structure itself.

19. Why can't you delete a parent row with children?

The FK constraint would leave orphans. Options: delete children first (my delete_event loops event.bookings), or ON DELETE CASCADE to auto-delete children, or SET NULL.

20. What is cascading / ON DELETE CASCADE?

FK rule: when the parent row is deleted, children are automatically deleted (or SET NULL etc.). SQLAlchemy: cascade='all, delete' on the relationship. I cascade manually in the route so the logic is visible.

SQL language

21. What is CRUD in SQL?

INSERT (Create), SELECT (Read), UPDATE, DELETE.

22. Select everything from user table?

SELECT * FROM user;

23. Names starting with S? (asked verbatim, repeatedly)

SELECT * FROM user WHERE name LIKE 'S%'; — % matches anything; '%S' = ending with S; '%S%' = containing S.

24. LIKE vs ILIKE?

LIKE is case-sensitive (in most DBs), ILIKE case-insensitive. SQLAlchemy's .ilike() gives case-insensitive matching portably — my searches use it.

25. Insert a row?

INSERT INTO event (name, location, available_slots) VALUES ('Music Night', 'Nepal', 10);

26. Update a value?

UPDATE event SET available_slots = 9 WHERE id = 1; — omit WHERE and EVERY row updates (classic trap answer).

27. Delete all users? What if FK constraint?

DELETE FROM user; — errors if bookings still reference users (FK constraint). Delete bookings first: real viva moment from the sheet.

28. Count rows? Filter + count?

SELECT COUNT(*) FROM booking; / SELECT COUNT(*) FROM booking WHERE status='Booked';

29. What is a JOIN?

Combining rows of two tables on a matching column: SELECT b.id, u.name, t.name FROM booking b JOIN user u ON b.user_id=u.id JOIN event t ON b.event_id=t.id; — the ORM does this for me via relationships (b.user.name).

30. ORDER BY / LIMIT?

SELECT * FROM event ORDER BY name LIMIT 5; ORM: Event.query.order_by(Event.name).limit(5).all()

31. Why is SQL so verbose? (odd but real question)

It's declarative English-like by design (1970s goal: non-programmers querying data) — you say WHAT you want, the engine decides HOW. The ORM hides that verbosity behind Python.

32. What is SQL injection? Are you protected?

Attack where user input is pasted into SQL text ("'; DROP TABLE user; --"). The ORM sends parameters separately from the query (parameterized queries), so input is data, never code. Plus Jinja autoescaping covers XSS on output.

SQLAlchemy / ORM

33. What is an ORM? Advantages/disadvantages?

Object-Relational Mapper: classes↔tables, objects↔rows, attributes↔columns; you write Python, it writes SQL. Pros: no hand SQL, injection-safe, DB-portable, objects fit app code. Cons: slight overhead, hides SQL, complex queries can fight it.

34. Which ORM? Is SQLAlchemy() a class or function?

SQLAlchemy via the Flask-SQLAlchemy extension. It's a class — db = SQLAlchemy() instantiates it; db is my handle to Model, Column, session.

35. What is db.Model?

The declarative base class; inheriting it maps my class to a table (name = class name lowercased) and grants .query and session powers.

36. Where is the database created in your code?

Config line SQLALCHEMY_DATABASE_URI names the file; db.init_app(app) binds SQLAlchemy to Flask; db.create_all() inside app context creates tables from the models; setup_database() also seeds the admin. Programmatically end to end — no DB Browser.

37. What is db.session? Why commit?

Transaction staging area. add()/delete() and attribute changes accumulate; commit() writes them atomically; rollback() discards. No commit = nothing persists.

38. get vs get_or_404 vs filter_by().first()?

get(pk): by primary key, None if absent. get_or_404: same but aborts with a 404 page — my edit/delete routes use it so bad IDs can't crash. filter_by(...).first(): first row matching conditions, None if none.

39. filter_by vs filter?

filter_by: keyword equality only (status='Open'). filter: full expressions (Event.available_slots > 0, .ilike(), & combinations).

40. How do you search case-insensitively in your app?

Event.query.filter(Event.location.ilike('%' + q + '%')) — substring, any case. Used in admin search and user location filter.

41. Write the ORM query: all open events with slots left.

Event.query.filter(Event.status == 'Open', Event.available_slots > 0).all()

42. Write the ORM insert: add a user (real live task).

u = User(username='x', password_hash=generate_password_hash('p'), name='X', role='user') then db.session.add(u); db.session.commit()

43. Raw SQL equivalent of Event.query.filter_by(status='Open').all()?

SELECT * FROM event WHERE status = 'Open'; (Proctors ask "what SQL does this line produce?")

44. What is db.relationship? Is it a column?

Not a column — a Python-side navigation property built on the FK: event.bookings gives the related Booking objects without writing a join.

45. What is backref? backref vs back_populates?

backref auto-creates the reverse attribute: relationship on User with backref='user' gives every Booking a .user. back_populates does the same but declared explicitly on both sides. backref = less code (mine); back_populates = more explicit.

46. What is lazy loading / lazy=True vs lazy='dynamic'?

Related rows load only when accessed (default True/'select') — event.bookings triggers its query at access time. 'dynamic' returns a query object you can filter further before loading. Opposite: eager loading fetches upfront via join.

47. What is uselist=False?

Makes a relationship return a single object instead of a list — how User↔StaffProfile becomes one-to-one.

48. Why is there both a staff_id column and a staff backref on Event?

staff_id is the actual FK stored in the table; event.staff is the ORM convenience giving the full User object (event.staff.name in my templates). Column = data, relationship = navigation.

49. You changed a model but the DB didn't change — why?

create_all only creates missing tables; it never ALTERs existing ones. Dev fix: delete instance/eventhub.db and restart (data lost). Production fix: migrations (Flask-Migrate/Alembic).

50. How would you view the data directly?

DB Browser for SQLite opening instance/eventhub.db (read/verify only — creation must stay programmatic), or python: sqlite3 CLI, or flask shell with User.query.all().
