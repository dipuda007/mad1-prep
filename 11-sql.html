Lesson 11: SQL & SQLite

Goal: understand what the ORM will do for you — because proctors ask you to write raw SQL ("query for names starting with S") even when your app never does.

What is a relational database?

Data stored in tables (rows + typed columns) that reference each other. SQLite is a tiny relational database engine that lives in a single file (eventhub.db) — no server to install, which is why the course mandates it. SQL is the language for talking to it.

"Why SQLite? SQLite vs MySQL?" — SQLite: serverless, zero-configuration, whole DB in one file, perfect for local single-user apps (and the only DB the project permits). MySQL: a separate server process, handles many concurrent users, needed for production scale.

"Structured vs unstructured data?" — structured: fixed rows/columns (SQL databases). Unstructured: no fixed schema — images, free text, documents (NoSQL like MongoDB stores semi-structured JSON).

The commands (CRUD)

-- CREATE TABLE: define structure
CREATE TABLE event (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    location TEXT,
    slots INTEGER
);

-- C: INSERT (create rows)
INSERT INTO event (name, location, slots) VALUES ('Music Night', 'Nepal', 10);

-- R: SELECT (read)
SELECT * FROM event;                          -- all rows, all columns
SELECT name, slots FROM event;                -- some columns
SELECT * FROM event WHERE slots > 0;          -- filter
SELECT * FROM event WHERE name LIKE 'S%';     -- names starting with S  (viva classic!)
SELECT COUNT(*) FROM event;                   -- how many rows

-- U: UPDATE (change)
UPDATE event SET slots = 9 WHERE id = 1;      -- ALWAYS use WHERE or you update every row

-- D: DELETE
DELETE FROM event WHERE id = 1;
"What is CRUD?" — Create, Read, Update, Delete: the four basic operations on data. SQL: INSERT/SELECT/UPDATE/DELETE. HTTP: POST/GET/PUT/DELETE. Your admin does full CRUD on events.

Keys — the most important concept

TermMeaning
Primary keycolumn that uniquely identifies each row (your id columns); cannot be NULL, cannot repeat
Foreign keycolumn holding another table's primary key, creating a relationship — booking.event_id points at event.id
Uniquevalues may not repeat but it's not the row identifier (your username)
NOT NULLvalue is compulsory (nullable=False in SQLAlchemy)
"Primary vs unique key?" — both forbid duplicates, but a table has exactly ONE primary key which identifies rows and can't be NULL; it can have many unique columns and (in SQL generally) unique columns can allow NULL. "Primary vs foreign key?" — primary identifies rows in its own table; foreign references a primary key of another table to link them.

"How would you convert one-to-many into many-to-many?" — introduce a junction table holding two foreign keys (e.g. event_id + staff_id), one row per pairing; remove the single FK column.

Terms that get asked cold

  • Constraint — a rule the DB enforces: PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, DEFAULT, CHECK.
  • Normalization — organizing tables to avoid repeating data (e.g. we store event_id in a booking, not the event's whole name/location — those live once in the event table).
  • TRUNCATE vs DROP vs DELETE — DELETE removes rows (can filter with WHERE); TRUNCATE empties the whole table fast (SQLite doesn't have it; use DELETE without WHERE); DROP removes the table itself, structure and all.
  • NULL — "no value". Not 0, not empty string. staff_id is NULL until admin assigns someone.
  • Why can't I delete a user who has bookings? — the foreign key constraint: bookings still reference that user. Delete/handle the child rows first (your delete-event route does exactly this), or configure cascade.
