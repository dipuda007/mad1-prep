1. Dashboard with search/filter

@app.route('/user')
def user_dashboard():
    user = current_user()
    if not user or user.role != 'user':
        return redirect(url_for('login'))

    # optional search filters from the GET form
    category = request.args.get('category', '')
    location = request.args.get('location', '')

    query = Event.query.filter_by(status='Open')
    if category:
        query = query.filter_by(category=category)
    if location:
        query = query.filter(Event.location.ilike('%' + location + '%'))
    events = query.all()

    my_bookings = Booking.query.filter_by(user_id=user.id).all()
    return render_template('user_dashboard.html', events=events, my_bookings=my_bookings,
                           category=category, location=location)

Explainable gold in here:

  • filter_by(status='Open') — users only ever see Open events ("allow users to view approved/open events").
  • The query is built up step by step — filters only applied if the user filled them. This is your "search and filter events based on category and location".
  • request.args not request.form — the filter form is GET.
  • my_bookings filtered by user_id=user.id — users see only their own history.

2. The booking route — three guards

@app.route('/user/book/<int:event_id>', methods=['POST'])
def book_event(event_id):
    user = current_user()
    if not user or user.role != 'user':
        return redirect(url_for('login'))

    event = Event.query.get_or_404(event_id)

    if event.status != 'Open':
        flash('This event is not open for booking.')
        return redirect(url_for('user_dashboard'))
    if event.available_slots <= 0:
        flash('No slots left on this event.')
        return redirect(url_for('user_dashboard'))

    already = Booking.query.filter_by(user_id=user.id, event_id=event.id, status='Booked').first()
    if already:
        flash('You have already booked this event.')
        return redirect(url_for('user_dashboard'))

    booking = Booking(user_id=user.id, event_id=event.id, booking_date=today())
    event.available_slots -= 1
    db.session.add(booking)
    db.session.commit()
    flash('Event booked!')
    return redirect(url_for('user_dashboard'))

Be able to recite the three guards without looking:

  1. Status guard — "allow booking only when event status is Open".
  2. Slots guard — "prevent overbooking beyond available slots".
  3. Duplicate guard — "prevent duplicate event bookings"; note it filters status='Booked', so a user who cancelled CAN book again (deliberate — mention it before the proctor finds it).

Then: create booking + decrement slot + one commit. Both changes land together — one transaction (a nice word to use out loud).

3. Backend vs frontend validation — free viva points

These guards are backend validation — they run in Flask, cannot be bypassed. The required, type="number", min="0" attributes in your forms are frontend validation — instant feedback, but a crafted request skips them. "Which is more important?" — backend; frontend is convenience, backend is security. Your project has both.
