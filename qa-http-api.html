Q&A: HTTP, REST, MVC & Web Concepts (46 questions)

HTTP

1. What is HTTP?

HyperText Transfer Protocol — the request/response language between browsers and servers. Text-based, stateless, runs over TCP (port 80; HTTPS 443).

2. HTTP vs HTTPS — which is more secure and why?

HTTPS = HTTP over TLS/SSL encryption: traffic can't be read or tampered with in transit, and certificates verify the server's identity. Local dev uses HTTP because traffic never leaves the machine.

3. What does stateless mean?

Each request is independent; the protocol carries no memory of previous requests. Login "memory" is added on top via cookies/sessions.

4. Explain the request-response cycle.

Browser builds a request (method, URL, headers, optional body) → server routes it to a handler (my Flask view function) → handler returns a response (status code, headers, body HTML/JSON) → browser renders it. Every click/submit repeats this.

5. Name the HTTP methods.

GET, POST, PUT, PATCH, DELETE (+ HEAD, OPTIONS). CRUD mapping: POST create, GET read, PUT/PATCH update, DELETE delete.

6. GET vs POST (the #1 question)?

GET: data in the URL query string, visible/bookmarkable/cacheable, length-limited, must not change server data. POST: data in the body, not shown in URL, no practical length limit, used for creating/changing data and credentials.

7. Which sends more data — GET or POST?

POST — the body has no practical size limit; URLs are limited (~2000 chars typical).

8. Can GET do POST's work?

Technically a server could change data on GET, but it violates HTTP semantics: GETs get cached, prefetched, crawled — data would change accidentally. Passwords would also land in URLs/history/logs. So: never.

9. POST vs PUT? Can PUT create? Can you use PUT for a form?

POST creates a new resource each time (not idempotent — twice = two records). PUT replaces/updates at a known URL and IS idempotent (twice = same result); it may create if the target doesn't exist. HTML forms only support GET/POST, so my updates go via POST; PUT would need JS/Postman.

10. PUT vs PATCH?

PUT replaces the whole resource; PATCH modifies only the sent fields.

11. What is idempotent?

Repeating the request leaves the same final state as doing it once. GET, PUT, DELETE are idempotent; POST is not.

12. When is POST triggered in your app?

Submitting any form with method="POST": register, login, profile save, event add/edit/delete, approve, blacklist, staff update, book, cancel.

13. Status codes — groups and the ones you see?

1xx info, 2xx success (200 OK, 201 Created), 3xx redirect (301 permanent, 302 my redirects, 304 cached), 4xx client error (400 bad request, 401 unauthenticated, 403 forbidden, 404 not found, 405 wrong method), 5xx server error (500 my bug, 501 not implemented, 503 unavailable).

14. Why 302 in your terminal after login?

redirect() returns 302 Found; the browser then GETs the target URL — so a successful login shows POST /login 302 followed by GET /admin 200.

15. What is 304?

Not Modified — the browser asked "has style.css changed?" and the server said "use your cached copy". Normal, not an error.

16. Parts of a URL?

protocol://host:port/path?query#fragment — http://127.0.0.1:5000/user?category=Easy. Query string carries GET form data as key=value& pairs.

17. What is localhost / 127.0.0.1?

The loopback address — always this machine. Server and browser both on your laptop during development and the viva demo.

18. What is a port?

Numbered channel letting one machine run many network services: web 80/443, Flask dev 5000. host:port identifies exactly which program gets the request.

19. What is DNS?

Domain Name System — translates names (google.com) to IP addresses. Localhost skips it via the hosts file.

20. Headers? Examples?

Key-value metadata on requests/responses: Content-Type (text/html, application/json), Cookie / Set-Cookie (my session travels here), Location (redirect target), User-Agent.

21. How is form data transported to the server?

GET: appended to the URL as the query string. POST: in the request body as name=value pairs (Content-Type application/x-www-form-urlencoded). Flask parses them into request.args / request.form.

REST & APIs

22. What is an API?

Application Programming Interface — a contract for programs to talk to each other. Web API: URLs returning data (JSON) instead of pages.

23. What is REST / a RESTful API?

REpresentational State Transfer — design style: resources as nouns in URLs (/api/events/5), HTTP verbs as actions (GET read, POST create, PUT update, DELETE remove), stateless requests, JSON responses. RESTful = follows these conventions.

24. What is JSON?

JavaScript Object Notation — lightweight text data format of key-value pairs and lists: {"name": "Music Night", "slots": 10}. Looks like Python dicts; universal API format.

25. Why no API in your project? How would you add one?

APIs are an optional milestone; core spec wants server-rendered Jinja pages. To add: routes like @app.route('/api/events') returning dicts/lists (Flask auto-JSONs dicts), or Flask-RESTful with Resource classes; test with Postman or curl.

26. Advantages/disadvantages of REST APIs?

Pros: frontend-independent (same API serves web/mobile), stateless = scalable, standard verbs. Cons: needs multiple round trips for complex screens, versioning pain, overkill for a small server-rendered app.

27. Can an API return a boolean?

Not bare in Flask — wrap in JSON: return {'ok': True}. JSON itself supports true/false values.

28. What is Postman / curl?

Tools for sending arbitrary HTTP requests (any verb, headers, body) to test APIs without a browser.

29. What is a webhook vs API? (occasional L2 stretch)

API: you call them when you want data. Webhook: they call your URL when an event happens (push vs pull).

Architecture

30. What is MVC? Map it to your project.

Model-View-Controller separation: Model = data layer (models.py), View = presentation (templates/), Controller = request logic (route functions in app.py). Flow: request → controller → queries model → passes data to view → HTML response.

31. Why use MVC?

Separation of concerns: change UI without touching data logic, test layers separately, standard structure others can navigate.

32. Client-server vs distributed architecture?

Client-server: central server, many clients — my app. Distributed: functionality/data spread across cooperating machines (microservices, blockchain). A distributed system has no single point of failure but far more complexity.

33. 2-tier vs 3-tier?

Tiers = separable layers. 2-tier: client ↔ database. 3-tier: presentation (browser) ↔ application (Flask) ↔ data (SQLite) — my app. MVC organizes the code; tiers describe the runtime layers.

34. What is a single page application (SPA)?

App loading one HTML page, then updating the DOM with JavaScript + API calls (React/Vue) — no full reloads. Mine is the opposite: multi-page app, every action returns fresh server-rendered HTML (JS is disallowed for core).

35. Horizontal vs vertical scaling?

Vertical: bigger machine (more RAM/CPU). Horizontal: more machines behind a load balancer. SQLite would be swapped for a server DB before scaling horizontally.

36. What is scalability? Concurrency?

Scalability: capacity to handle growth in users/data. Concurrency: multiple things happening at once — the Flask dev server is single-threaded-ish and SQLite locks on writes: fine locally, replaced in production.

37. What is caching?

Storing results to avoid recomputing/refetching — browser caches static files (the 304s), servers cache pages/queries, CDNs cache globally.

38. Frontend vs backend?

Frontend: what runs in the browser — HTML/CSS (+ JS normally). Backend: server-side logic + database — Flask/SQLite. They meet at the HTTP request.

Security & validation

39. Authentication vs authorization (THE most-asked question)?

Authentication: proving WHO you are — my login's check_password_hash. Authorization: what you're ALLOWED to do — my role checks (user.role != 'admin' → redirect) and ownership checks (event.staff_id != user.id). Point to both blocks in code.

40. What security mechanisms does your app have?

Hashed+salted passwords (werkzeug), signed session cookie (SECRET_KEY), role checks on every route, ownership checks (staff/own bookings), staff approval gate, blacklist gate at login, ORM = SQL-injection safe, Jinja autoescaping = XSS-safe, POST for all state changes.

41. Frontend vs backend validation — which matters more?

Frontend (required, type=number, min): instant UX, but bypassable. Backend (my status/slots/duplicate guards, username-taken check): authoritative — requests can be forged, so the server must re-check everything. Backend wins; have both.

42. What validations exist in your project?

Frontend: required fields, number min=0, date pickers, dropdowns constraining values. Backend: unique username, wrong-password rejection, blacklist/approval gates, event Open check, slots > 0 check, duplicate-booking check, staff-ownership check, status whitelist, int() conversions.

43. What is hashing vs encryption? Why different hashes for the same password?

Encryption reversible with a key; hashing one-way. Random salt is mixed per user and stored in the hash string — same password, different salt → different hash; defeats rainbow tables.

44. What is XSS / CSRF (stretch questions)?

XSS: injecting scripts into pages viewed by others — Jinja autoescaping defends. CSRF: tricking a logged-in browser into submitting a forged request — defended by CSRF tokens (Flask-WTF adds them); my simple app relies on session checks, and I can name the token approach as the improvement.

45. If a user reports a bug, how do you help them? (real question)

Reproduce it: get steps/screenshot, run the same flow locally, watch the terminal traceback and status codes, check the DB state, fix, test the flow again, deploy. Methodical answer > specific answer here.

46. What are cookies used for besides sessions?

Preferences (theme/language), analytics/tracking, "remember me" tokens. Stored per-domain, sent automatically with each request to that domain, size-limited (~4KB).
