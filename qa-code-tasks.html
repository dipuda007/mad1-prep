Live Code Tasks — practiced solutions (25 tasks)

⚠️ Note: examples on this page reference EventHub, the demo app built in this course — not any term's official project. Map every pattern to YOUR statement and never copy code into your submission (plagiarism = U grade).

Every viva includes "change your code so that…". These are the real ones from past vivas, translated to the event-booking app. Practice each once before the viva. Golden rules: think out loud, edit the right layer (template for looks, route for logic, model for schema), restart the server after .py changes (templates only need refresh), and if a model changed — delete instance/eventhub.db.

Warm-ups they ask cold (often in Notepad)

1. Write a Flask route returning "Hello world".

@app.route('/hello')
def hello():
    return 'Hello world'

2. Route /course/<name> printing "course is [name]".

@app.route('/course/<course_name>')
def course(course_name):
    return 'course is ' + course_name

3. Jinja for loop over a list, only items > 1.

{% for n in a %}{% if n > 1 %}{{ n }}{% endif %}{% endfor %}

4. Form page + route: input text → submit → next page shows the text.

@app.route('/echo', methods=['GET', 'POST'])
def echo():
    if request.method == 'POST':
        return 'You typed: ' + request.form['text']
    return '<form method="POST"><input name="text"><button>Go</button></form>'
(Returning inline HTML is fine for a live task; mention a template is the proper way.)

5. Two number inputs → page showing their sum.

Same shape: total = int(request.form['a']) + int(request.form['b']), return f-string. Remember int()!

6. Flask endpoint reading URL params ?name=x&course=y returning JSON.

@app.route('/info')
def info():
    return {'name': request.args.get('name'),
            'course': request.args.get('course')}

7. SQL: users whose name starts with S / ORM version.

SELECT * FROM user WHERE name LIKE 'S%'; — ORM: User.query.filter(User.name.like('S%')).all()

8. ORM: fetch all users / add a row to a table.

User.query.all(). Add: create object → db.session.add(obj) → db.session.commit().

UI changes (template/CSS layer)

9. Change the background colour.

static/style.css → body { background-color: #eef; } → save → hard refresh (Ctrl+F5, CSS may be cached). Inline alternative: style="background-color:..." on body in base.html.

10. Background image (+ gradient) on login page.

In style.css: body { background-image: url('mountain.jpg'); background-size: cover; } (file placed in static/). With darkening gradient: background-image: linear-gradient(rgba(0,0,0,.4), rgba(0,0,0,.4)), url('mountain.jpg');

11. Move a button (e.g. Register link / Add Event) to top-right.

HTML order + Bootstrap: my Add Event already sits in a d-flex justify-content-between header row. Generic answer: wrap in <div class="text-end"> or add class float-end, or physically move the tag above the table.

12. Stack the dashboard stat cards vertically instead of a row.

In admin_dashboard.html change every col-md-3 to col-12 (each takes the full 12-unit row). Explaining the 12-column grid while doing it = extra points.

13. Swap two navbar items / move a link to the end.

base.html — reorder the <a class="nav-link"> lines; the last one before </div> renders rightmost. ms-auto on the group pushes the whole block right.

14. Bold a table column.

Wrap the cell content: <td><strong>{{ event.name }}</strong></td> — or CSS td:nth-child(2) { font-weight: bold; }.

15. Add a "Reset/Clear" button to the login form (classic!).

One line inside the form: <button type="reset" class="btn btn-outline-secondary">Clear</button> — type="reset" empties fields, pure HTML, no JS.

Logic changes (route layer)

16. Limit username/password length (e.g. max 8–10 chars).

register(): after reading the form, if len(username) > 10: flash('Max 10 characters'); return redirect(url_for('register')). Also show the frontend twin: maxlength="10". They usually want backend.

17. Username must contain @ / password needs special char + capital.

if '@' not in username: flash(...). Password: if not any(c.isupper() for c in password) or not any(c in '!@#$%' for c in password): flash(...)

18. Confirm-password field.

Template: second input name="confirm". Route: if request.form['password'] != request.form['confirm']: flash('Passwords do not match'); return redirect(url_for('register'))

19. Add a phone/age field to user registration that lands in the DB (very common!).

Three layers, say them in order: (1) models.py User: age = db.Column(db.Integer); (2) register.html: <input type="number" name="age">; (3) register(): age=int(request.form['age']) in the User(...) call. Then DELETE instance/eventhub.db + restart (create_all won't alter the existing table) — knowing this step is what they're testing. Display: add a column in admin_users.html.

20. Limit total events to N (admin can't add more than 5).

Top of add_event POST: if Event.query.count() >= 5: flash('Limit of 5 events reached'); return redirect(url_for('admin_dashboard'))

21. Prevent a user having more than 2 active bookings total.

In book_event before creating: if Booking.query.filter_by(user_id=user.id, status='Booked').count() >= 2: flash('Booking limit reached'); return redirect(url_for('user_dashboard'))

22. Filter a listing by a value (only events with slots == X / price = X).

Route: events = Event.query.filter_by(available_slots=int(request.args.get('slots'))).all() or comparison via .filter(). Pattern matters: read param → filter → pass to template.

23. New page: dropdown of all events from the DB + search shows bookings for it.

Route GET: fetch events for the dropdown; on submit read event_id from request.args; query Booking.query.filter_by(event_id=...); render results. Template: <select name="event_id">{% for t in events %}<option value="{{ t.id }}">{{ t.name }}</option>{% endfor %}</select> — dynamic dropdown = loop over query results.

24. Add a search by another field (e.g. events by category in admin search).

Extend admin_search: events_by_diff = Event.query.filter(Event.category.ilike('%'+q+'%')).all() and merge/display. Shows you can extend your own pattern — exactly what they check.

25. "What happens if admin deletes an event that has bookings?" — then make it safer.

Current behaviour: bookings are deleted with it (manual cascade). Safer version live: block deletion when active bookings exist — active = Booking.query.filter_by(event_id=event.id, status='Booked').count(); if active > 0: flash('Cancel active bookings first'); return redirect(...). Or set event.status='Closed' instead of deleting (soft delete). Explaining the trade-off = L2 points.

If the change doesn't show: 1) .py change? → restart server (Ctrl+C, python app.py). 2) model change? → delete instance/eventhub.db too. 3) CSS change? → hard refresh Ctrl+F5. 4) still weird? → close and reopen VS Code (a real student's viva was saved by exactly this).
