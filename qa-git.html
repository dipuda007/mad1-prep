Q&A: Git & Version Control (20 questions)

1. What is a version control system (VCS)? Another name?

Software recording changes to files over time so you can recall any version. Also called source control / revision control. Git is the dominant one.

2. What is Git?

A distributed VCS: snapshots ("commits") of your project with full history on every machine. Created by Linus Torvalds for Linux kernel development.

3. Git vs GitHub?

Git = the tool on your machine. GitHub = a website hosting git repositories for backup/collaboration. Alternatives to GitHub: GitLab, Bitbucket.

4. Is git centralized or distributed? Why is distributed better?

Distributed — every clone holds complete history; you commit/browse history offline; no single point of failure. Centralized VCS (SVN) needs the server for almost everything.

5. Uses of git besides tracking changes?

Collaboration (branches/merges), experimenting safely, reverting mistakes, code review via pull requests, and here: my commit history is evidence the project is my own work built over time.

6. What is a repository?

A project folder tracked by git — the .git subfolder holds all history. Local repo on my machine, remote repo on GitHub.

7. What is a commit? A good commit message?

A saved snapshot with author, timestamp, message. Message should say what/why concisely — my milestone commits use the required unique messages like "Milestone-3 TMA-MAD-1 Admin dashboard and management".

8. What does git add do? The staging area?

add moves changes into the staging area (index) — the "shopping cart" of what the next commit will contain. git add . stages everything; commit then snapshots the staged set.

9. The everyday workflow?

git status (see changes) → git add . (stage) → git commit -m "msg" (snapshot) → git push (upload to GitHub).

10. What is git push / pull / clone / fetch?

push: upload local commits to the remote. pull: download + merge remote commits. clone: copy an entire remote repo (with history) to your machine. fetch: download without merging.

11. What is a branch? Create one? (asked as a live task)

An independent line of development. git branch feature-x creates; git checkout feature-x switches; git checkout -b feature-x does both. Default branch: main.

12. What is a merge? A merge conflict?

merge brings another branch's commits into the current one. Conflict = both branches edited the same lines; git marks the file and you choose manually, then commit.

13. What is .gitignore?

A file listing path patterns git must not track. Mine: __pycache__/, instance/, venv/, *.pyc — machine-generated files don't belong in history.

14. Stop tracking a file but keep it on disk? (asked verbatim)

git rm --cached filename, then add it to .gitignore, commit.

15. See history? Undo things?

History: git log (--oneline for short). Undo unstaged file changes: git checkout -- file (or git restore file). Unstage: git reset file. Revert a pushed commit safely: git revert <hash> (makes an opposite commit).

16. What is git status vs git log?

status: current uncommitted state (what changed, what's staged). log: the commit history.

17. What is a remote? What is origin?

A named URL of a hosted copy. origin = default name for the remote you cloned from / added — mine points to github.com/24f2008594/MAD-1-PROJECT-IITM.

18. How did GitHub authentication work from your terminal?

Git Credential Manager (bundled with Git for Windows) pops a browser login on first push, then stores a token in Windows Credential Manager — later pushes don't ask. Switching accounts = remove the credential and push again.

19. Why is your repo private with a collaborator?

Course rule: code must not be public (plagiarism risk — both parties get U); MADI-cs2003 added as collaborator so the team can track milestone commits. That's Milestone 0.

20. What is a pull request?

GitHub feature: propose a branch's changes for review before merging — how teams review code. Solo project: I commit straight to main.
