Lesson 12: Flask-SQLAlchemy — the ORM

Goal: the #1 viva topic. Every proctor asks "what is ORM". Own this lesson.

What is an ORM?

Object-Relational Mapper: a translator between Python objects and database tables. Classes ↔ tables, objects ↔ rows, attributes ↔ columns. You write Python; the ORM writes the SQL.

You write (Python)ORM runs (SQL)
Event.query.all()SELECT * FROM event;
Event.query.get(5)SELECT * FROM event WHERE id=5;
db.session.add(t); db.session.commit()INSERT INTO event VALUES (...);
t.slots = 9; db.session.commit()UPDATE event SET slots=9 WHERE id=...;
"ORM advantages/disadvantages?" — Advantages: no hand-written SQL, protection against SQL injection, database-independent (swap SQLite→MySQL by changing one config line), results are Python objects. Disadvantage: slightly slower than raw SQL, hides what's happening, and very complex queries can be awkward. "Which ORM did you use?" — SQLAlchemy, via the Flask-SQLAlchemy extension. "Is SQLAlchemy() a class or function?" — a class; db = SQLAlchemy() creates an instance.

Setup (exactly as your project does it)

# models.py
from flask_sqlalchemy import SQLAlchemy
db = SQLAlchemy()

# app.py
from models import db
app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///eventhub.db'
db.init_app(app)
"Where is your database defined/initialized?" — the URI config line in app.py says which database (a SQLite file named eventhub.db, created inside the instance/ folder); db.init_app(app) connects SQLAlchemy to the Flask app; db.create_all() creates the tables from the model classes.

Defining a model

class Event(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(100), nullable=False)
    available_slots = db.Column(db.Integer, nullable=False)
    staff_id = db.Column(db.Integer, db.ForeignKey('user.id'))

Column types: db.Integer, db.String(n), db.Boolean, db.Float, db.Date… Options: primary_key=True, unique=True, nullable=False, default=.... Table name defaults to the class name lowercased → FK strings are lowercase: db.ForeignKey('user.id').

Queries — your app's vocabulary

Event.query.all()                          # list of all events
Event.query.get(3)                         # one event by primary key (or None)
Event.query.get_or_404(3)                  # same but auto-404 page if missing
Event.query.filter_by(status='Open').all() # WHERE status='Open'
Event.query.filter_by(status='Open').first()   # first match or None
Event.query.count()                        # number of rows
User.query.filter_by(username='ravi').first() # login lookup
Event.query.filter(Event.location.ilike('%nepal%')).all()  # case-insensitive search
"like vs ilike?" — like is case-sensitive pattern match, ilike is case-insensitive; % means "anything" — '%nep%' matches Nepal, NEPAL, event-booking-in-nepal. "filter_by vs filter?" — filter_by takes simple keyword equality (filter_by(status='Open')); filter takes full expressions (filter(Event.slots > 0), ilike, comparisons).

Changing data — the session

# CREATE
t = Event(name='Music Night', available_slots=10, ...)
db.session.add(t)
db.session.commit()

# UPDATE — just change the object
t.available_slots -= 1
db.session.commit()

# DELETE
db.session.delete(t)
db.session.commit()
"What is db.session? Why commit?" — the session is a staging area collecting your changes (a transaction); nothing touches the database file until commit() writes them all at once. Forgot commit = changes vanish. That's the answer to "I added a row, why isn't it in the DB?"

Relationships & backref — asked in EVERY viva

class Event(db.Model):
    ...
    bookings = db.relationship('Booking', backref='event')

class Booking(db.Model):
    ...
    event_id = db.Column(db.Integer, db.ForeignKey('event.id'), nullable=False)

The FK column creates the link in the database. db.relationship creates Python conveniences on BOTH sides:

  • some_event.bookings → list of that event's Booking objects
  • some_booking.event → the Event object it belongs to (this reverse attribute is what backref creates)
"backref vs back_populates?" — both wire up the two-way link. backref: declared on ONE side, the other side is generated automatically (what your project uses — less code). back_populates: you declare the relationship explicitly on BOTH classes and name each side — more typing, more explicit. Functionally equivalent here.

"What is lazy loading / lazy=True?" — related rows are NOT fetched when you load the parent; SQLAlchemy runs the extra query only at the moment you access event.bookings. Default behaviour. (lazy='dynamic' would give a query object you can filter further instead of a list.)

"What is cascading / cascade='all, delete'?" — rules for what happens to child rows when the parent changes/is deleted; with delete-cascade, deleting an event would auto-delete its bookings. Your project deletes the bookings explicitly in the route instead — same result, visible logic.

"uselist=False?" — makes the relationship return one object instead of a list — that's how User ↔ StaffProfile becomes one-to-one.
