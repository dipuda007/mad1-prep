Lesson 7: How the Web Works

Goal: the mental model behind everything — and one of the heaviest viva areas. No typing this lesson; read twice.

Client and server

The client is the browser. The server is a program waiting for requests — in your project, Flask. The conversation is always:

Browser (client)                     Flask (server)
      |  ---- HTTP REQUEST  --->          |
      |  "GET /login"                     |   runs your login() function
      |  <--- HTTP RESPONSE ----          |
      |  HTML + status code 200           |
renders the HTML

This is the request–response cycle. The server never contacts the browser first; it only answers.

"Is HTTP stateless? What does that mean?" — yes. Every request is independent; the server keeps no memory of previous requests by itself. That's why logins need sessions/cookies (Lesson 13) to "remember" who you are between requests.

HTTP methods (verbs)

MethodMeaningIn your project
GETfetch a page; data (if any) rides in the URL ?q=everestviewing dashboards, search forms
POSTsend data to create/change something; data hidden in request bodyregister, login, book, delete forms
PUTreplace/update a resource fully (idempotent)not used — HTML forms only support GET/POST
PATCHpartial update—
DELETEremove a resource— (we POST to a /delete route instead)
"GET vs POST?" — GET carries data in the URL (visible, bookmarkable, cached, length-limited, never for passwords) and should not change data on the server. POST carries data in the request body (not visible in URL, no length limit, right for creating/changing data).

"POST vs PUT?" — POST creates (sending twice creates two records); PUT updates/replaces and is idempotent — sending the same PUT twice leaves the same result as once.

"Why did you use POST for update/delete instead of PUT/DELETE?" — plain HTML forms can only send GET and POST; PUT/DELETE are used in APIs (e.g. via JavaScript or Postman), and my core requirements forbid JS.

HTTP vs HTTPS

HTTPS = HTTP + encryption (TLS/SSL). Data between browser and server is unreadable to anyone in between. Real sites use HTTPS; your local dev server uses HTTP because it never leaves your machine.

What is an API / REST? (asked even if you didn't build one)

An API endpoint is a URL that returns data (usually JSON) instead of an HTML page, so other programs can use it. REST is a style for designing such APIs: resources live at nouns (/api/events), and HTTP verbs express the action — GET reads, POST creates, PUT updates, DELETE removes. JSON is text that looks like a Python dict: {"name": "Music Night", "slots": 10}. In Flask, returning a dict from a route automatically becomes JSON. If asked "why no API in your app?" — it's an optional milestone; core requirements are server-rendered HTML through Jinja templates.

"Client-server vs distributed architecture?" — client-server: many clients talk to one central server (your app). Distributed: work/data spread across many machines cooperating (big systems like Google). Related term "3-tier": presentation (browser) → application (Flask) → data (SQLite) — your app is exactly 3-tier.
