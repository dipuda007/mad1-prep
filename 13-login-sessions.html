Lesson 13: Login & Sessions

Goal: how "being logged in" works. Last theory lesson before you build.

The problem

HTTP is stateless — the server forgets you after every request. So how does the app know request #2 comes from the same logged-in Ravi as request #1?

Sessions + cookies

Flask's session is a dictionary that survives between requests per visitor:

from flask import session

# at login:
session['user_id'] = user.id
session['role'] = user.role

# later, any route:
if 'user_id' in session: ...      # someone is logged in

# at logout:
session.clear()

Under the hood Flask stores this dict in a cookie (a small piece of data the browser keeps and re-sends with every request). The cookie is signed with your SECRET_KEY so users can't tamper with it.

"What is the secret key for? What breaks without it?" — it cryptographically signs the session cookie (and flash messages use the session too). Without it, Flask raises an error the moment you use session or flash. If the key changes, all existing sessions become invalid — everyone is logged out. "Cookies vs session?" — a cookie is browser-side storage sent with each request; the session is the server's concept of per-user data, which Flask happens to store inside a signed cookie. "Ways to store data client-side?" — cookies, localStorage, sessionStorage (the latter two via JS).

Password hashing — never store plain passwords

from werkzeug.security import generate_password_hash, check_password_hash

# at register:
user.password_hash = generate_password_hash('secret123')
# stores something like: scrypt:32768:8:1$Kj2...  — cannot be reversed

# at login:
check_password_hash(user.password_hash, typed_password)   # True/False
"Hashing vs encryption?" — encryption is reversible (with a key you get the original back); hashing is one-way — you can never recover the password from the hash. Login works by hashing the typed password and comparing hashes. "Why do two users with the same password have different hashes?" — a random salt is mixed in before hashing and stored inside the hash string; same input + different salt = different hash. This also defeats precomputed rainbow-table attacks.

Role-based access control (RBAC)

Each user row carries a role. After login you route them to their own dashboard, and every protected route re-checks the role:

@app.route('/admin')
def admin_dashboard():
    user = current_user()
    if not user or user.role != 'admin':
        return redirect(url_for('login'))
    ...
"Authentication vs authorization?" — THE most asked viva question. Authentication = verifying who you are (username + password check at login). Authorization = verifying what you're allowed to do (role check at the top of each route). In your code: authentication is the check_password_hash block in login(); authorization is the user.role != 'admin' guard in every admin route. Point at both. "What is RBAC?" — role-based access control: permissions attached to roles (admin/staff/user), users get the permissions of their role.

flash() — one-time messages

from flask import flash
flash('Event booked!')                 # store message in session
return redirect(url_for('user_dashboard'))
<!-- base.html shows and empties the stored messages -->
{% for message in get_flashed_messages() %}
    <div class="alert alert-info">{{ message }}</div>
{% endfor %}

Flash rides on the session (hence needs SECRET_KEY) and disappears after being displayed once.

Why not Flask-Login? Flask-Login is an extension that automates this (login_user(), @login_required, UserMixin, current_user). It's an optional milestone. Plain sessions are fewer moving parts and easier to defend line-by-line in a viva. If asked: "I implemented sessions directly to show I understand the mechanism Flask-Login abstracts away."
